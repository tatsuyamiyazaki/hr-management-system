# SDD Validate — 検証

## 前提
本スキルの実行前に以下を読み込むこと:
- `../sdd-core/SKILL.md`（全体ルール）

モードに応じて追加で読み込む:
- **gap**: `../sdd-core/references/rules/gap-analysis.md`
- **design**: `../sdd-core/references/rules/design-review.md`
- **impl**: `../sdd-core/references/rules/design-principles.md`

## 3つの検証モード

---

### Mode 1: Gap Analysis（ギャップ分析）
**トリガー**: `validate-gap` / 「ギャップ分析」
**タイミング**: 既存プロジェクトに新機能を追加する前（要件生成後、設計前）
**前提条件**: requirements.md が生成済み

**プロセス**:
1. **現状調査**: コードベースのスキャン、規約抽出、統合サーフェス特定
2. **要件実現可能性分析**: 技術ニーズのリスト化、ギャップと制約の特定
3. **実装アプローチ選択肢**:
   - オプションA: 既存コンポーネント拡張
   - オプションB: 新規コンポーネント作成
   - オプションC: ハイブリッド
4. **工数・リスク評価**: S/M/L/XL × 高/中/低

**出力**: ユーザーへのインタラクティブなレポート（選択肢とトレードオフ提示）

---

### Mode 2: Design Review（設計レビュー）
**トリガー**: `validate-design` / 「設計レビュー」
**タイミング**: design.md 生成後、タスク生成前
**前提条件**: design.md が生成済み

**プロセス**:
1. **分析**: 全レビュー基準に対して設計を評価
   - 既存アーキテクチャとの整合性
   - 設計の一貫性と標準
   - 拡張性と保守性
   - 型安全とインターフェース設計
2. **重要課題の特定**: 最大3つ（課題、影響、提案、トレーサビリティ）
3. **強みの認識**: 1-2の良い点
4. **GO/NO-GO判定**: 根拠付き

**出力**: 設計レビューサマリー + インタラクティブディスカッション

---

### Mode 3: Implementation Validation（実装検証）
**トリガー**: `validate-impl` / 「実装検証」
**タイミング**: 実装完了後
**前提条件**: 実装コードが存在

**プロセス**:
1. **要件カバレッジ**: requirements.md の全受け入れ基準が実装されているか
2. **設計準拠**: design.md のインターフェース・コンポーネント境界に従っているか
3. **品質チェック**:
   - 型安全（`any` 不使用等）
   - エラーハンドリングの網羅性
   - テストカバレッジ
   - ステアリング規約への準拠
4. **ギャップレポート**: 未実装の要件、設計との乖離、品質問題をリスト化

**出力**: 実装検証レポート（カバレッジマトリクス + 問題一覧 + 推奨アクション）

---

## 共通ルール
- 検証結果は常にユーザーに対話的に提示
- 問題の修正は自動で行わない（レポートと推奨のみ）
- 検証後の承認/修正判断はユーザーに委ねる
