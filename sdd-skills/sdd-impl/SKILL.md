# SDD Impl — TDD実装

## 前提
本スキルの実行前に以下を読み込むこと:
- `../sdd-core/SKILL.md`（全体ルール）

## 概要
承認済みのタスクに基づき、TDD（テスト駆動開発）方式でコードを実装する。
タスク完了時にtasks.mdのチェックボックスを更新する。

## 前提条件
- `ready_for_implementation == true`（全フェーズ承認済み）
- `.kiro/specs/{feature}/tasks.md` が存在

## 入力
- **feature名**: 対象の仕様ディレクトリ名
- **tasks**（オプション）: 実装するタスク番号の指定（例: `1.1`, `1.1-1.3`, `2`）
  - 未指定時: 未完了の最初のタスクから順に実装

## 実行プロセス

### Step 1: コンテキスト読み込み
1. `.kiro/specs/{feature}/tasks.md` — 実装タスク一覧
2. `.kiro/specs/{feature}/design.md` — アーキテクチャ・インターフェース定義
3. `.kiro/specs/{feature}/requirements.md` — 受け入れ基準
4. `.kiro/specs/{feature}/research.md` — 技術的判断・制約
5. `.kiro/steering/` — プロジェクト標準・規約

### Step 2: タスク選択
- 指定されたタスク番号、または未完了の最初のタスクを選択
- 並列マーカー `(P)` を考慮: 並列タスクは同時実行の提案が可能

### Step 3: TDD実装サイクル
各サブタスクに対して:

```
1. Red   — 受け入れ基準に基づくテストを書く（失敗を確認）
2. Green — テストを通す最小限のコードを書く
3. Refactor — コード品質を改善（テスト維持）
```

**実装ルール**:
- design.md のインターフェース定義に従う
- ステアリングの技術標準・命名規約に従う
- 型安全を維持（TypeScriptでは `any` 禁止）
- エラーハンドリングを実装（design.md のError Handling参照）

### Step 4: チェックボックス更新
タスク完了時に tasks.md を更新:
```markdown
- [x] 1.1 (P) 完了したサブタスク
```

### Step 5: 進捗報告
実装完了後にユーザーに報告:
- 完了したタスク
- 作成/変更したファイル
- テスト結果
- 次のタスク

## タスク番号の指定方法
- 単一: `1.1` — サブタスク1.1のみ
- 範囲: `1.1-1.3` — サブタスク1.1から1.3まで
- メジャー: `2` — メジャータスク2の全サブタスク
- 複数: `1.1, 2.1, 3` — 指定したタスク全て

## 実装中の問題対応

### 設計との乖離
design.md と実装で矛盾が生じた場合:
1. design.md の意図を優先的に確認
2. 軽微な調整 → 実装で対応し、design.md にメモ追加
3. 重大な乖離 → ユーザーに報告、設計変更の承認を得る

### 要件の曖昧さ
受け入れ基準の解釈が不明確な場合:
1. requirements.md を再確認
2. 最も合理的な解釈で実装
3. ユーザーに解釈を報告し確認

### 予期しない技術的問題
1. research.md に記録
2. 影響範囲を評価
3. ユーザーに報告し対応方針を決定

## 品質チェックリスト（各タスク完了時）
- [ ] テストが通過しているか
- [ ] design.md のインターフェースに準拠しているか
- [ ] ステアリングの規約に従っているか
- [ ] エラーハンドリングが実装されているか
- [ ] tasks.md のチェックボックスが更新されているか
