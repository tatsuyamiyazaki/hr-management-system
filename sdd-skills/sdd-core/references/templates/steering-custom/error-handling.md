# エラーハンドリング標準

## 哲学
- Fail Fast: 無効な状態を早期に検出して拒否
- 回復可能なエラーと致命的なエラーを区別
- エラーメッセージは開発者とユーザーの両方に有用

## エラー分類

| カテゴリ | 例 | 戦略 |
|---------|-----|------|
| バリデーション | 無効入力、型不一致 | 即座に拒否、フィールド別メッセージ |
| ビジネスルール | 残高不足、権限不足 | 条件を説明、代替手段を提示 |
| インフラ | DB接続断、タイムアウト | リトライ、フォールバック、グレースフル劣化 |
| 外部サービス | API障害、レート制限 | サーキットブレーカー、キャッシュ |
| プログラミング | null参照、未処理例外 | ログ記録、汎用メッセージ返却 |

## エラー表現
```typescript
// 判別共用体パターン（推奨）
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

// カスタムエラークラス
class AppError extends Error {
  constructor(
    public code: string,
    message: string,
    public statusCode: number,
    public context?: Record<string, unknown>
  ) {
    super(message);
  }
}
```

## ロギング
- エラーレベル: ERROR（要対応）, WARN（注意）, INFO（追跡用）
- 構造化ログ: JSON形式、相関ID付与
- 機密情報のマスキング必須
- スタックトレースは開発環境のみ

## リトライ戦略
- 一時的障害のみリトライ（冪等な操作に限定）
- 指数バックオフ + ジッター
- 最大リトライ回数の設定
- リトライ不可のエラーを明示的に定義

## ユーザー向けエラー
- 技術的詳細を隠す
- アクション可能なメッセージを提供（「～してください」形式）
- エラーコードで分類（サポート問い合わせ時の参照用）

---
_エラーハンドリングのパターンと方針を記述。全エラーコード一覧ではない。_
