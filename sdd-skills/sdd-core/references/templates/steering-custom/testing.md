# テスト標準

## テスト哲学
- テストは仕様であり、ドキュメントである
- 高速フィードバックループを優先
- 信頼性の低いテストは即修正または削除

## テストピラミッド

| レイヤー | 割合 | スコープ | 速度 |
|---------|------|---------|------|
| Unit | 70% | 関数/モジュール単体 | ms |
| Integration | 20% | コンポーネント間連携 | sec |
| E2E | 10% | ユーザーフロー全体 | sec-min |

## 命名規約
```
[テスト対象]_[条件/シナリオ]_[期待結果]
```
例: `createUser_duplicateEmail_throwsConflictError`

## テスト構造（AAA）
```
// Arrange — 前提条件のセットアップ
// Act — テスト対象の実行
// Assert — 結果の検証
```

## モック/スタブ方針
- 外部サービス・IO → モック必須
- 内部ロジック → 可能な限り実物を使用
- モックの過剰使用は設計の問題を隠す

## カバレッジ目標
- 新規コード: [例: 80%以上]
- クリティカルパス: [例: 95%以上]
- カバレッジは品質の指標ではなく最低ラインの指標

## CI統合
- PRマージ前にテスト全通過必須
- フレーキーテストのトラッキングと対応プロセス
