# デプロイメント標準

## デプロイ哲学
- 頻繁に、小さく、安全にデプロイ
- 自動化を前提（手動ステップの最小化）
- ロールバック可能な状態を常に維持

## 環境

| 環境 | 用途 | デプロイトリガー |
|------|------|----------------|
| Development | ローカル開発 | - |
| Staging | 統合テスト・QA | PR マージ / 手動 |
| Production | 本番サービス | リリースタグ / 承認後 |

## CI/CD パイプライン
```
コミット → Lint/Format → テスト → ビルド → [Staging デプロイ → E2E] → Production デプロイ
```

## デプロイ戦略
- **方式**: [Blue-Green / Canary / Rolling / 直接]
- **ヘルスチェック**: デプロイ後の自動検証
- **ロールバック**: [自動 / 手動 / 条件付き]

## 環境変数・設定管理
- シークレット: [例: AWS Secrets Manager, Vault]
- 設定: [例: 環境変数, .env ファイル（ローカルのみ）]
- 環境間の設定差分の管理方法

## コンテナ化（該当する場合）
- ベースイメージ: [例: node:20-slim]
- マルチステージビルド使用
- イメージサイズの最小化
- レジストリ: [例: ECR, GCR, Docker Hub]

## モニタリング・アラート
- メトリクス: [例: レスポンスタイム、エラーレート、CPU/メモリ]
- ロギング: [例: CloudWatch, Datadog]
- アラート条件: [例: エラーレート > 1%、レイテンシ p99 > 2s]
- オンコール体制（該当する場合）

## データベースマイグレーション
- デプロイ前にマイグレーション実行
- 後方互換性のあるマイグレーション（ゼロダウンタイム）
- マイグレーション失敗時のロールバック手順
