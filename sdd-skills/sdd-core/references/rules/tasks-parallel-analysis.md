# 並列タスク分析ルール

## 目的
`tasks.md` 生成時に、安全に並列実行できる実装タスクを一貫して特定するための基準を提供する。

## 並列と判断する条件
以下の**全て**が真の場合にのみタスクを並列実行可能としてマーク:

1. **データ依存なし**: 他の未完了タスクからのデータに依存しない
2. **共有リソースの競合なし**: 同じ変更可能ファイルや共有リソースに触れない
3. **前提レビュー/承認不要**: 他タスクからの事前レビューや承認を必要としない
4. **環境/セットアップ充足**: タスクに必要な環境やセットアップが既に満たされている、またはタスク内で完結する

## マーキング規約
- 条件を満たすタスクの数値IDの直後に `(P)` を付与
  - 例: `- [ ] 2.1 (P) メール送信用バックグラウンドワーカーの構築`
- メジャータスクとサブタスクの両方に適用可能
- 順次実行が要求された場合（`--sequential` フラグ等）は `(P)` マーカーを完全に省略
- `(P)` はチェックボックスブラケットの**外側**に配置（完了状態との混同を避ける）

## グループ化と順序のガイドライン
- 並列タスクは同一テーマの場合、同じ親の下にグループ化
- 詳細箇条書きで前提条件や注意事項を記載（例: 「1.2のスキーママイグレーション完了が必要」）
- 類似タスクが並列安全でない場合は、ブロッキング依存関係を明示的に記載
- コンテナのみのメジャータスク（独自のアクション可能な詳細箇条書きがないもの）には `(P)` をマークしない — サブタスクレベルで並列実行を評価

## 品質チェックリスト
タスクに `(P)` をマークする前に以下を確認:

- このタスクを並行実行してもマージやデプロイのコンフリクトが発生しないこと
- 共有状態の期待がある場合は詳細箇条書きに記載していること
- 実装が独立してテスト可能であること

いずれかのチェックが失敗した場合は `(P)` を**マークせず**、タスク詳細で依存関係を説明する。
