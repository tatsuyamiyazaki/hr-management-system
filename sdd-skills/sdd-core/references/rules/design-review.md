# 設計レビュープロセス

## 目的
技術設計文書のインタラクティブな品質レビューを実施し、許容可能なリスクで実装に進められることを確認する。

## レビュー哲学
- **品質保証であり、完璧さの追求ではない**
- **重要な問題に集中**: 最大3つの最重要懸念に限定
- **インタラクティブな対話**: 一方的な評価ではなく設計者との対話
- **バランスの取れた評価**: 強みと弱みの両方を認識
- **明確な判定**: 根拠を伴う確定的なGO/NO-GO判定

## コアレビュー基準

### 1. 既存アーキテクチャとの整合性（重要）
- 既存システムの境界とレイヤーとの統合
- 確立されたアーキテクチャパターンとの一貫性
- 適切な依存方向と結合管理
- 現在のモジュール構成との整合

### 2. 設計の一貫性と標準
- プロジェクトの命名規約とコード標準への準拠
- 一貫したエラーハンドリングとロギング戦略
- 統一された設定と依存関係管理
- 確立されたデータモデリングパターンとの整合

### 3. 拡張性と保守性
- 将来の要件に対する設計の柔軟性
- 関心の分離と単一責任の明確化
- テスト容易性とデバッグへの配慮
- 要件に対する適切な複雑さ

### 4. 型安全とインターフェース設計
- 適切な型定義とインターフェースコントラクト
- unsafe パターンの回避（TypeScriptでの `any` 等）
- 明確なAPI境界とデータ構造
- 入力検証とエラーハンドリングのカバレッジ

## レビュープロセス

### Step 1: 分析
全レビュー基準に対して設計を分析し、統合、保守性、複雑さ、要件充足に影響する重要な問題に集中。

### Step 2: 重要課題の特定（3つ以下）
各課題について:
```
🔴 **重要課題 [1-3]**: [簡潔なタイトル]
**懸念**: [具体的な問題]
**影響**: [なぜ重要か]
**提案**: [具体的な改善策]
**トレーサビリティ**: [要件ID/requirements.mdのセクション]
**根拠**: [design.mdのセクション/見出し]
```

### Step 3: 強みの認識
バランスの取れたフィードバック維持のため1-2の良い点を認識。

### Step 4: GO/NO-GO判定
- **GO**: 重大なアーキテクチャ不整合なし、要件対処済み、明確な実装パス、許容可能なリスク
- **NO-GO**: 根本的な矛盾、重大なギャップ、高い失敗リスク、不均衡な複雑さ

## 出力フォーマット

### 設計レビューサマリー
全体的な品質と準備状況について2-3文。

### 重要課題（3つ以下）
各課題: 課題、影響、推奨、トレーサビリティ（例: 1.1, 1.2）、根拠（design.mdセクション）。

### 設計の強み
1-2の良い点。

### 最終評価
判定（GO/NO-GO）、根拠（1-2文）、次のステップ。

### インタラクティブディスカッション
設計者の視点、代替案、明確化、必要な変更について対話。
