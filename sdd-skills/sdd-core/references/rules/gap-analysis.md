# ギャップ分析プロセス

## 目的
要件と既存コードベースのギャップを分析し、実装戦略の判断に必要な情報を提供する。

## 分析フレームワーク

### 1. 現状調査

- ドメイン関連アセットのスキャン:
  - 主要ファイル/モジュールとディレクトリレイアウト
  - 再利用可能なコンポーネント/サービス/ユーティリティ
  - 支配的なアーキテクチャパターンと制約

- 規約の抽出:
  - 命名、レイヤリング、依存方向
  - import/exportパターンと依存ホットスポット
  - テスト配置とアプローチ

- 統合サーフェスの特定:
  - データモデル/スキーマ、APIクライアント、認証メカニズム

### 2. 要件実現可能性分析

- EARS要件から技術的ニーズをリスト化:
  - データモデル、API/サービス、UI/コンポーネント
  - ビジネスルール/バリデーション
  - 非機能: セキュリティ、パフォーマンス、スケーラビリティ、信頼性

- ギャップと制約の特定:
  - 現在のコードベースに不足している機能
  - 後で調査が必要な未知の項目（「要リサーチ」としてマーク）
  - 既存アーキテクチャからの制約

- 複雑度シグナルの特定:
  - シンプルなCRUD / アルゴリズムロジック / ワークフロー / 外部統合

### 3. 実装アプローチ選択肢

#### オプションA: 既存コンポーネントの拡張
**検討タイミング**: 機能が既存構造に自然にフィットする場合

- **拡張するファイル/モジュール**: 具体的な変更対象ファイル、既存機能への影響、後方互換性
- **互換性評価**: 既存インターフェースの尊重、消費者への破壊的変更なし
- **複雑さと保守性**: 認知負荷、単一責任原則の維持

**トレードオフ**:
- ✅ 最小限の新ファイル、初期開発速度
- ✅ 既存パターンとインフラの活用
- ❌ 既存コンポーネントの肥大化リスク
- ❌ 既存ロジックの複雑化の可能性

#### オプションB: 新規コンポーネント作成
**検討タイミング**: 機能が明確な責務を持つ、または既存コンポーネントが既に複雑な場合

- **新規作成の根拠**: 関心の明確な分離、既存コンポーネントの複雑さ、独立したライフサイクル
- **統合ポイント**: 既存システムとの接続方法、公開APIやインターフェース
- **責務境界**: 新コンポーネントの所有範囲、データフローと制御フロー

**トレードオフ**:
- ✅ 関心の明確な分離
- ✅ 独立したテストの容易さ
- ❌ ナビゲートするファイル数の増加
- ❌ 慎重なインターフェース設計が必要

#### オプションC: ハイブリッドアプローチ
**検討タイミング**: 拡張と新規作成の両方が必要な複雑な機能

- **組み合わせ戦略**: どの部分を拡張し、どの部分を新規作成するか
- **段階的実装**: 初期フェーズ（最小限の変更）→ 後続フェーズ（リファクタ/新規作成）
- **リスク緩和**: インクリメンタルなロールアウト、フィーチャーフラグ

### 4. スコープ外

- 詳細なリサーチ活動は設計フェーズに延期
- 不明点は簡潔な「要リサーチ」項目としてのみ記録

### 5. 実装の複雑さとリスク

- 工数:
  - S (1-3日): 既存パターン、最小限の依存、直接的な統合
  - M (3-7日): 新パターン/統合あり、中程度の複雑さ
  - L (1-2週間): 重要な機能、複数の統合やワークフロー
  - XL (2週間以上): アーキテクチャ変更、不慣れな技術、広範な影響

- リスク:
  - 高: 未知の技術、複雑な統合、アーキテクチャシフト
  - 中: ガイダンス付きの新パターン、管理可能な統合
  - 低: 確立されたパターンの拡張、親しみのある技術

### 出力チェックリスト

- 要件-アセットマップ（ギャップタグ付き: 不足 / 不明 / 制約）
- オプションA/B/Cの簡潔な根拠とトレードオフ
- 工数(S/M/L/XL)とリスク(高/中/低)の1行説明
- 設計フェーズへの推奨事項: 推奨アプローチ、主要決定事項、リサーチ項目

## 原則

- **判断より情報**: 分析と選択肢を提供し、最終決定は下さない
- **複数の実行可能な選択肢**: 適用可能な場合は信頼できる代替案を提示
- **明示的なギャップと前提**: 不明点と制約を明確にフラグ
- **コンテキスト認識**: 既存パターンとアーキテクチャの限界に沿う
- **透明な工数とリスク**: ラベルを簡潔に正当化
