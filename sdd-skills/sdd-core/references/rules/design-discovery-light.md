# ライトディスカバリープロセス（拡張向け）

## 目的
機能拡張のための既存システムと統合要件を迅速に分析する。

## 集中ディスカバリーステップ

### 1. 拡張ポイント分析
**統合アプローチの特定**:
- 既存の拡張ポイントやインターフェースを特定
- 変更スコープ（ファイル、コンポーネント）を決定
- 従うべき既存パターンを確認
- 後方互換性要件を特定

### 2. 依存関係チェック
**互換性の検証**:
- 新規依存関係のバージョン互換性を確認
- APIコントラクトが変更されていないか検証
- パイプラインに破壊的変更がないことを確認

### 3. クイック技術検証
**新規ライブラリのみ**:
- WebSearchで公式ドキュメントを検索
- 基本的な使用パターンを確認
- 既知の互換性問題を確認
- ライセンス互換性を確認
- 主要な発見を `research.md`（技術整合性セクション）に記録

### 4. 統合リスク評価
**クイックリスクチェック**:
- 既存機能への影響
- パフォーマンスへの影響
- セキュリティ考慮事項
- テスト要件

## フルディスカバリーへのエスカレーション条件
以下を発見した場合はフルディスカバリーに切り替え:
- 重大なアーキテクチャ変更が必要
- 複雑な外部サービス統合
- セキュリティに敏感な実装
- パフォーマンスクリティカルなコンポーネント
- 不明または文書化不十分な依存関係

## 出力要件
- 明確な統合アプローチ（`research.md`に境界への影響を記録）
- 変更するファイル/コンポーネントのリスト
- バージョン付きの新規依存関係
- 統合リスクと緩和策
- テストの重点分野
