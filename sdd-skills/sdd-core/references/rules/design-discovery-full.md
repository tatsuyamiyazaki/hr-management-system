# フルディスカバリープロセス

## 目的
技術設計が完全で正確かつ最新の情報に基づくことを保証するための包括的な調査・分析を実施する。

## ディスカバリーステップ

### 1. 要件分析
**要件から技術的ニーズを抽出**
- EARS形式の全機能要件を抽出
- 非機能要件（パフォーマンス、セキュリティ、スケーラビリティ）を特定
- 技術的制約と依存関係を決定
- コアとなる技術的課題をリスト化

### 2. 既存実装分析
**現在のシステムを理解**（修正/拡張の場合）:
- コードベースの構造とアーキテクチャパターンを分析
- 再利用可能なコンポーネント、サービス、ユーティリティをマッピング
- ドメイン境界とデータフローを特定
- 統合ポイントと依存関係を文書化
- アプローチ決定: 拡張 vs リファクタ vs ラップ

### 3. 技術リサーチ
**ベストプラクティスと解決策の調査**:
- **WebSearch**で調査:
  - 類似問題に対する最新のアーキテクチャパターン
  - 技術スタックに関する業界ベストプラクティス
  - 関連技術の最新アップデートや変更
  - 一般的な落とし穴と解決策

- **WebFetch**で分析:
  - フレームワーク/ライブラリの公式ドキュメント
  - APIリファレンスと使用例
  - マイグレーションガイドと破壊的変更
  - パフォーマンスベンチマークと比較

### 4. 外部依存関係の調査
**各外部サービス/ライブラリについて**:
- 公式ドキュメントとGitHubリポジトリを検索
- APIシグネチャと認証方法を検証
- 既存スタックとのバージョン互換性を確認
- レート制限と使用制約を調査
- コミュニティリソースと既知の問題を発見
- セキュリティ考慮事項を文書化
- 実装調査が必要なギャップを記録

### 5. アーキテクチャパターン・境界分析
**アーキテクチャオプションの評価**:
- 関連パターンの比較（MVC, Clean, Hexagonal, Event-driven）
- 既存アーキテクチャとステアリング原則との適合性評価
- チームコンフリクトを避けるためのドメイン境界と所有権の特定
- スケーラビリティへの影響と運用上の懸念を考慮
- 保守性とチーム専門知識を評価
- 推奨パターンと却下された代替案を `research.md` に記録

### 6. リスク評価
**技術的リスクの特定**:
- パフォーマンスボトルネックとスケーリング限界
- セキュリティ脆弱性と攻撃ベクター
- 統合の複雑さと結合度
- 技術的負債の創出 vs 解消
- 知識ギャップとトレーニングの必要性

## リサーチガイドライン

### 検索すべきタイミング
**常に検索**:
- 外部APIドキュメントと更新
- 認証/認可のセキュリティベストプラクティス
- 特定されたボトルネックのパフォーマンス最適化手法
- 依存関係の最新バージョンとマイグレーションパス

**不確かな場合に検索**:
- 特定ユースケースのアーキテクチャパターン
- データフォーマット/プロトコルの業界標準
- コンプライアンス要件（GDPR, HIPAA等）
- 予想負荷に対するスケーラビリティアプローチ

### 検索戦略
1. 公式ソース（ドキュメント、GitHub）から開始
2. 最近のブログ記事と記事を確認（直近6ヶ月）
3. Stack Overflowで一般的な問題を確認
4. 類似のオープンソース実装を調査

## 出力要件
設計判断に影響する全ての発見を `research.md` に共有テンプレートを使用して記録:
- アーキテクチャ、技術整合性、コントラクトに影響する主要な洞察
- リサーチ中に発見された制約
- 推奨アプローチと選択されたアーキテクチャパターン（根拠付き）
- 却下された代替案とトレードオフ
- コンポーネント・インターフェースコントラクトに反映する更新されたドメイン境界
- リスクと緩和策
- 実装中にさらなる調査が必要なギャップ
