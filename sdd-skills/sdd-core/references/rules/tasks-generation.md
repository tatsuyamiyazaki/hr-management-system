# タスク生成ルール

## 基本原則

### 1. 自然言語での記述
機能と成果に焦点を当て、コード構造ではない。

**記述すべき**:
- 達成すべき機能
- ビジネスロジックと振る舞い
- フィーチャーと能力
- ドメイン言語と概念
- データ関係とワークフロー

**避けるべき**:
- ファイルパスとディレクトリ構造
- 関数/メソッド名とシグネチャ
- 型定義とインターフェース
- クラス名とAPIコントラクト
- 具体的なデータ構造

**根拠**: 実装詳細（ファイル、メソッド、型）はdesign.mdで定義。タスクは実行すべき機能的作業を記述する。

### 2. タスクの統合と進行

**全タスクは**:
- 前のタスクの出力に基づいて構築（孤立コードなし）
- システム全体に接続（ぶら下がりフィーチャーなし）
- インクリメンタルに進行（大きなジャンプなし）
- コア機能を早期に検証
- design.mdのアーキテクチャ境界を尊重
- design.mdのインターフェースコントラクトを遵守
- メジャータスクのサマリーは控えめに — サブタスクで完全にカバーされる場合は詳細箇条書きを省略

**統合タスクで締めくくり**、全体を接続する。

### 3. 柔軟なタスクサイズ

**ガイドライン**:
- **メジャータスク**: 論理的に必要な数のサブタスク（凝集性でグループ化）
- **サブタスク**: 各1-3時間、3-10の詳細項目
- 細かすぎず粗すぎないバランス

**任意の数を強制しない** — 論理的なグループ化で構造を決定。

### 4. 要件マッピング

**各タスク詳細セクションの末尾に**:
- `_Requirements: X.X, Y.Y_` として**数値の要件IDのみ**をカンマ区切りで記載。説明テキスト、括弧、翻訳、自由形式のラベルは付けない
- 横断的要件の場合は関連する全要件IDを列挙
- 全要件はrequirements.mdで数値IDを持つ必須。IDがない場合は停止してrequirements.mdを修正

### 5. コードのみに集中

**含めるもの**:
- コーディングタスク（実装）
- テストタスク（ユニット、統合、E2E）
- 技術セットアップタスク（インフラ、設定）

**除外**:
- デプロイタスク
- ドキュメントタスク
- ユーザーテスト
- マーケティング/ビジネス活動

### オプションのテストカバレッジタスク

- 設計が機能カバレッジを保証しMVPの迅速な配信が優先される場合、純粋にテスト指向のフォローアップ作業を `- [ ]*` チェックボックスで**オプション**としてマーク
- 受け入れ基準をサブタスクの詳細箇条書きで直接参照する場合のみオプションマーカーを適用
- 実装作業や統合クリティカルな検証をオプションにしない — `*` は補助的/延期可能なテストカバレッジに限定

## タスク階層ルール

### 最大2レベル
- **レベル1**: メジャータスク（1, 2, 3, 4...）
- **レベル2**: サブタスク（1.1, 1.2, 2.1, 2.2...）
- **より深いネストなし**（1.1.1は不可）
- メジャータスクが単一のアクション可能項目しか含まない場合、構造を折りたたみサブタスクをメジャーレベルに昇格
- メジャータスクが純粋なコンテナとして存在する場合、チェックボックスの説明は簡潔にし、詳細はサブタスクに委譲

### 連続番号
- メジャータスクは必ずインクリメント: 1, 2, 3, 4, 5...
- サブタスクはメジャータスクごとにリセット: 1.1, 1.2, then 2.1, 2.2...
- メジャータスク番号の重複禁止

### 並列分析（デフォルト）
- 明示的に無効化されない限り並列分析は有効
- **全条件**が満たされる場合に並行実行可能なタスクを特定:
  - 他の未完了タスクへのデータ依存なし
  - 共有ファイルやリソースの競合なし
  - 他タスクからの前提レビュー/承認不要
- タスク番号の直後に `(P)` を付与: `- [ ] 2.1 (P) バックグラウンドワーカーの構築`
- メジャータスクとサブタスクの両方に適用可能
- 順次モードが要求された場合は `(P)` マーカーを完全に省略
- 並列でないタスクには依存関係を明示的に記載

### チェックボックス形式
```markdown
- [ ] 1. メジャータスクの説明
- [ ] 1.1 サブタスクの説明
  - 詳細項目1
  - 詳細項目2
  - _Requirements: X.X_

- [ ] 1.2 サブタスクの説明
  - 詳細項目...
  - _Requirements: Y.Y_

- [ ] 2. 次のメジャータスク（1ではなく2！）
- [ ] 2.1 サブタスク...
```

## 要件カバレッジ

**必須チェック**:
- requirements.mdの全要件がカバーされていること
- 全要件IDとタスクマッピングをクロスリファレンス
- ギャップ発見時: 要件または設計フェーズに戻る
- 対応するタスクのない要件は許容しない

意図的に延期する要件がある場合は根拠を記録。
